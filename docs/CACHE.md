# 视频处理缓存功能

## 概述

为了避免重复处理同一个视频，系统实现了智能缓存机制。当用户再次发送已经处理过的视频链接时，系统会使用缓存的文本内容重新创建飞书文档，无需重新下载、识别和精转。

## 功能特点

- **自动缓存**：每次成功处理视频后，自动保存文本内容到缓存
- **快速响应**：同一视频再次处理时，直接使用缓存文本（跳过下载、识别、精转流程）
- **重新创建文档**：每次都创建新的飞书文档，获得新的文档链接
- **持久化存储**：缓存保存在本地文件，程序重启后依然有效
- **完整信息**：缓存包含原始文本、精转文本、关键纪要等完整内容

## 工作原理

### 缓存检查流程

```
用户发送视频链接
    ↓
提取视频ID
    ↓
检查缓存是否存在？
    ├─ 是 → 使用缓存文本 → 重新创建文档 ✅
    └─ 否 → 正常处理流程 → 保存文本到缓存
```

## 技术实现

### 缓存模块

- **文件位置**：`src/cache_utils.py`
- **缓存文件**：`temp/cache/video_cache.json`
- **实现方式**：JSON文件存储 + 单例模式

### 核心API

```python
from cache_utils import get_cache

# 获取缓存实例
cache = get_cache()

# 检查视频是否已缓存
if cache.exists(video_id):
    # 获取缓存数据（包含文本内容）
    cached_data = cache.get(video_id)
    original_text = cached_data['original_text']
    refined_text = cached_data['refined_text']
    summary_text = cached_data['summary_text']
    # 使用这些文本重新创建文档...

# 保存处理结果到缓存（保存文本内容）
cache.set(
    video_id=video_id,
    video_title=video_title,
    original_text=original_text,
    refined_text=refined_text,
    summary_text=summary_text
)
```

### 集成到任务流程

在 `src/task.py` 的 `process_video_sync` 函数中：

1. **步骤0**：检查缓存（新增）
2. **步骤1-8**：正常处理流程
3. **步骤9**：保存到缓存（新增）

## 性能优势

### 处理时间对比

| 场景 | 无缓存 | 有缓存 | 提升 |
|------|--------|--------|------|
| 首次处理 | 3-7分钟 | 3-7分钟 | 无变化 |
| 再次处理 | 3-7分钟 | **<5秒** | **36-84倍** |

### 节省时间分析

使用缓存时，跳过以下耗时步骤：
- ✅ 视频下载：节省 1-2秒（音频）或 5-10分钟（视频）
- ✅ 音频提取：节省 5-10秒
- ✅ 语音识别：节省 1-3分钟
- ✅ 文本精转：节省 1-2分钟
- ✅ 关键纪要生成：节省 1-2分钟

仅保留：
- 📝 创建飞书文档：约 3-5秒


```python

```

## 为什么缓存文本而非文档链接？

### 优势

1. **灵活性**：每次可以创建新文档，获得新的分享链接
2. **更新性**：如果文档模板改进，可以重新生成文档
3. **可控性**：可以控制文档的访问权限和有效期
4. **一致性**：确保所有文档使用最新的格式和样式

### 对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| **缓存文本**（当前） | 每次创建新文档，灵活性高 | 每次需要调用文档API（3-5秒） |
| 缓存文档链接 | 直接返回，速度极快 | 文档固定，无法更新格式 |

### 选择建议

- 如果您的视频需要多次分享 → 使用缓存文本（当前方案）
- 如果您追求极致速度 → 可以改为缓存文档链接

## 注意事项

1. **缓存文件位置**：`temp/cache/video_cache.json`
2. **自动忽略**：`.gitignore` 已包含 `temp/`，缓存不会被提交到版本控制
3. **持久化**：程序重启后缓存依然有效
4. **唯一标识**：使用视频ID（BV号/AV号）作为缓存键
